🧭 [context] why does protocol choice matter?

In distributed systems, the protocol stack sits on the critical path of every request. The choice between TCP, REST, and gRPC is not merely an API preference — it is a latency and throughput contract with the operating system, the kernel scheduler, and the network stack.

The Problem: Most protocol comparisons report only end-to-end latency under a single load condition. They do not explain why protocols differ, nor do they show how overhead evolves with message size and concurrency. Without this, protocol selection is intuition-driven rather than evidence-based.

Design principle — uniform serialization: All three protocols used protobuf serialization, including raw TCP. This deliberately isolates protocol overhead as the only variable. Any latency difference is attributable to framing, flow control, and multiplexing — not encoding cost.

why Go over C++?
Go provides production-quality libraries for all three protocols under one consistent runtime — same GC, same scheduler, same allocator. Protocol differences are not contaminated by implementation quality gaps. Go's M:N goroutine model also maps 500 goroutines to ~8 OS threads at high concurrency, keeping client-side scheduling noise lower than 500 raw pthreads would in C++. Tradeoff: Go runtime overhead is inseparable from protocol overhead. A C++ baseline is left as future work.
[protocol_profiles]
TCP (raw)
  • Manual 4-byte length-prefix framing
  • No application-level flow control
  • One persistent connection per client
  • Zero protocol state management
  • All work done in kernel space
→ Fastest average. Worst tail. Best CPU efficiency.
REST (HTTP/1.1)
  • Persistent connections (net/http)
  • Binary protobuf body — not JSON
  • One goroutine per connection
  • Connection pool with mutex locking
  • 6 user-space layers before kernel
→ 2× TCP overhead. Scales poorly under concurrency.
gRPC (HTTP/2)
  • Stream multiplexing over one connection
  • Built-in flow control (WINDOW_UPDATE)
  • HPACK header compression
  • Binary 9-byte frame headers
  • 8+ user-space layers before kernel
→ Best tail latency. Best under stress. Highest baseline cost.

protocol inversion effect
Low concurrency, small messages: TCP (18µs) < REST (35µs) < gRPC (47µs)

The performance ranking completely inverts. The fastest protocol at average case has the worst tail latency at scale.

Why gRPC wins: HTTP/2 WINDOW_UPDATE frames explicitly limit in-flight unacknowledged data. TCP and HTTP/1.1 have no equivalent mechanism — senders blast data regardless of receiver capacity, causing kernel socket buffer saturation and catastrophic queue buildup.

TCP — shallow, kernel-dominated
2–3 user-space frames before kernel:
main.sendRecv → syscall.write → vfs_write → sock_write_iter → inet_sendmsg → tcp_sendmsg → ip_queue_xmit → __do_softirq

No futex calls visible anywhere. Zero synchronization overhead. Most CPU time is the kernel doing actual network work.
REST — deep HTTP/1.1 pipeline
6 user-space frames before kernel:
main.sendRecv → (*Client).Post → (*Transport).dialConn → persistConnWrite → bufio.Flush → net.(*conn).Write → syscall.write

Each layer reads/writes HTTP header buffers — explains the 6× cache reference increase. futex_wait calls visible — connection pool locking per request.
gRPC — distributed CPU across subsystems
8+ user-space frames. CPU time spread across: HTTP/2 transport, protobuf, stream management, flow control, keepalive pings — each visible as distinct towers. Multiple futex call sites for stream table locking.

[anomalies_and_root_causes]

Three anomalies were investigated at kernel level. Each was traced to a confirmed root cause rather than accepted as unexplained variance.

🔬 Anomaly 1: Page fault step function at 32KB — Go allocator boundary

Observed: Page fault count spiked discontinuously at 32KB across all gRPC configurations — a step function, not a gradual increase.

At 31KB — all three frame sizes
gRPC-8KB: ~4,117 page faults
gRPC-16KB: ~4,106 page faults
gRPC-32KB: ~3,394 page faults

→ Low and stable across all variants
At 32KB — all three frame sizes
gRPC-8KB: ~16,828 page faults
gRPC-16KB: ~14,043 page faults
gRPC-32KB: ~16,081 page faults

→ 4× simultaneous jump in all variants
confirmed root cause — Go allocator
Go runtime constant: _MaxSmallSize = 32768 (runtime/malloc.go)

Below → thread-local size-class pool → page already mapped → no fault.
Above → large-object allocator → fresh heap page from OS → guaranteed fault per allocation.
alternative hypothesis ruled out
HTTP/2 frame splitting (default 16KB) was initially suspected. Three gRPC builds were tested — 8KB, 16KB, and 32KB frame sizes. All three showed identical step functions at exactly 32KB across two independent runs. Frame size had zero effect — definitively ruling out HTTP/2 framing.
📊 Anomaly 2: Three-zone buffer behavior across message sizes

Observed: REST/gRPC latency ordering is not monotonic. Page fault analysis across 512B–64KB revealed three distinct performance zones.

Zone 1: <2KB
REST faster
Zone 2: 2–31KB
gRPC faster
Zone 3: ≥32KB
REST faster again
ZoneSize RangeRoot Cause
Zone 1<2KBgRPC HTTP/2 fixed setup overhead (stream tables, HPACK, flow control) generates more page faults than REST's simple handling

Zone 2→3 crossover pinpointed between 31KB and 32KB with 1KB resolution. Confirmed across two independent runs.

⚠️ Anomaly 3: REST latency exceeds gRPC at 4KB

At 4KB, c=1: REST (55.4µs) > gRPC (50.4µs) — reversing expected ordering. Reproduced in multiple runs. This is the Zone 1→2 crossover manifesting in latency.

explanation
Below ~1KB, gRPC's fixed HTTP/2 setup allocations generate more page faults than REST. Above 2KB, REST's dynamic bufio growth overtakes gRPC's stable pre-allocated buffers. [discussion]
🧠 The abstraction cost — quantified

REST over HTTP/1.1 adds 6 user-space function call layers, 6× more memory accesses, 3× more context switches, and 36× more mutex lock operations compared to raw TCP for identical 64B payloads. gRPC adds even more complexity per-request (2.7× more CPU cycles) but its multiplexing and flow control prevent catastrophic degradation at scale.

key insight
There is no universally best protocol. The ranking depends entirely on workload regime:

Low concurrency, small messages → TCP (latency, CPU efficiency)
Medium concurrency, predictable load → REST (simplicity, 2× overhead acceptable)
High concurrency, tail latency SLAs → gRPC (only defensible choice)
⚖️ Limitations and future work
  • Client-side profiling only: All perf measurements were on the client process. Server-side profiling would reveal receive-path overhead independently.
  • Go runtime overhead: GC and goroutine scheduler add overhead inseparable from protocol overhead. A C++ baseline would isolate language runtime costs.
  • Symmetric echo design: Asymmetric designs (small ACK only, or small ping with large response) would isolate send-path vs receive-path overhead independently.
  • Go allocator threshold: Modifying _MaxSmallSize to shift the 32KB boundary requires recompiling the Go runtime — the definitive confirmation of the allocator hypothesis.
  • Single-node only: Network latency would dominate µs-level differences in a distributed setup. Relative ratios and tail latency patterns remain applicable.
[when_to_use]
Use TCP when:
  • You control both endpoints
  • Payloads are small and uniform
  • Concurrency is bounded (<50)
  • Message size stays under 32KB
  • Minimum median latency is critical
HFT order entry, internal hot paths, game servers
Use REST when:
  • Simplicity and debuggability matter
  • Concurrency is low (<50)
  • Broad client compatibility needed
  • Human-readable debugging required
  • 2× TCP overhead is acceptable
Public APIs, low-traffic internal services
Use gRPC when:
  • P99 tail latency SLAs matter
  • Concurrency is high (>50)
  • Payloads vary in size
  • Streaming RPCs needed
  • Internal microservice mesh
Latency-sensitive distributed systems, ML inference
systems design rule
Average latency is what your system does on a good day.
P99 latency is what your system does to your users.
→ Optimize for tail, not mean. TCP fails this test catastrophically at scale.
