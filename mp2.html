Triangular Interleaver (5G-NR)

  • Achieved ~0.41µs latency (~25× faster than baseline)
  • Converted triangular mapping into BRAM-based permutation lookup
  • Parallel processing (16 bits/cycle) with pipelined execution
[problem]

The standard triangular interleaver algorithm requires writing data row-wise into a triangular matrix and reading it out column-wise. In software, this is trivial. In physical hardware, this leads to:

  • Irregular memory access patterns that break cache locality.
  • Complex nested loop structures requiring heavy mathematical division and modulo operations.
  • Massive state-machine overhead resulting in multi-microsecond latency.
[key insight]

The mathematical structure of a triangular matrix for a fixed block size is static and completely deterministic. Instead of forcing the FPGA to compute complex indices at runtime (wasting DSP slices and clock cycles), the math can be bypassed entirely.

The Fix: Precompute the entire physical permutation mapping at compile-time, store it in Block RAM (BRAM), and convert complex runtime logic into a pure O(1) memory lookup.

[design]
  • Precomputed permutation table synthesized directly into FPGA BRAM.
  • Aggressive loop unrolling: processing 16 elements per clock cycle instead of 1.
  • Fully pipelined AXI4-Stream architecture to guarantee zero-stall throughput.
[results]

Latency: ~0.41µs (for 512 bits)
Throughput: 16 elements per cycle
Improvement: ~25× over standard algorithmic implementation

🔬 Deep Dive: Why the naive approach is slow

In a naive C++ implementation, calculating the exact read/write index for a triangular structure requires solving quadratic equations or running deeply nested for loops.

When High-Level Synthesis (HLS) compiles nested loops with dynamic variables, it cannot synthesize a straight wire. Instead, it builds a massive Finite State Machine (FSM). For every single bit of data, the FPGA has to stall, calculate the math using ALUs and DSP slices, update the state, and then move the data. This creates a severe pipeline bottleneck where the math calculation takes longer than the actual data movement.

⚙️ Deep Dive: Lookup-based optimization

By shifting from a "Compute" paradigm to a "Memory" paradigm, we exploit the FPGA's physical architecture. The compile-time array is mapped directly to dual-port Block RAM (BRAM).

When the 512-bit payload enters the AXI-Stream bus, the hardware no longer asks, "What is the mathematical root of this index?" Instead, it asks the BRAM, "Where does bit 42 go?" The BRAM responds in exactly 1 clock cycle. By unrolling the loop by a factor of 16, we query the BRAM 16 times concurrently, saturating the bus and achieving wire-speed latency.

📊 Trade-offs

Silicon Area vs. Speed: This architecture intentionally trades logic area for pure execution speed. By precomputing the indices, we consume more BRAM tiles on the FPGA die. However, we simultaneously free up Look-Up Tables (LUTs) and DSP slices that would have been wasted on mathematical address generation.

In latency-critical environments like 5G baseband processing or streaming architectures with latency constraints, memory space is relatively cheap since the data is predominantly in-flight between/within the IPs, but nanoseconds are priceless. Thus, trading BRAM to save nanoseconds is mathematically optimal for the use case.
