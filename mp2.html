1. Software Concept: The Triangular Matrix (Row-In, Col-Out)

DATA IN (ROW) --> [ D0 D1 D2 D3 ] [ D4 D5 D6 ] [ D7 D8 ] [ D9 ] | v DATA OUT (COL) <-- [ D0 D4 D7 D9 D1 D5 D8 D2 D6 D3 ]

2. Hardware Architecture: O(1) Memory Lookup Path

[AXI-STREAM IN] ----> [ INPUT BUFFER ] ----> [ PERMUTATION LOGIC ] ----> [AXI-STREAM OUT] | ^ | (READ ADDR) | +------------------------+ | [ DUAL-PORT BRAM ] (LUT: f(i) -> addr)
[problem]

The standard triangular interleaver algorithm requires writing data row-wise into a triangular matrix and reading it out column-wise. In physical hardware, this leads to:

  • Irregular memory access patterns that break cache locality.
  • Complex nested loop structures requiring heavy mathematical division and modulo operations.
  • Massive state-machine overhead resulting in multi-microsecond latency.
[key_insight]

The mathematical structure of a triangular matrix for a fixed block size is static and completely deterministic. Instead of computing f(i) at runtime, we store pre-calculated perm[i] and route directly via dual-port BRAM. This converts an O(N) math problem into a wire-speed O(1) lookup.

[design]
  • Memory Mapping: Precomputed permutation table synthesized directly into FPGA BRAM.
  • Loop Unrolling: Aggressive unrolling to process 16 elements per clock cycle instead of 1.
  • Pipeline: Fully pipelined AXI4-Stream architecture to guarantee zero-stall throughput.

🔬 Deep Dive: Why the naive approach is slow

In a naive C++ implementation, calculating the exact read/write index for a triangular structure requires solving quadratic equations or running deeply nested for loops.

When High-Level Synthesis (HLS) compiles nested loops with dynamic variables, it cannot synthesize a straight wire. Instead, it builds a massive Finite State Machine (FSM). For every single bit of data, the FPGA has to stall, calculate the math using ALUs and DSP slices, and update the state. This creates a severe pipeline bottleneck where the math calculation takes longer than the actual data movement.

⚙️ Deep Dive: Lookup-based optimization

By shifting from a "Compute" paradigm to a "Memory" paradigm, we exploit the FPGA's physical architecture. The compile-time array is mapped directly to dual-port Block RAM (BRAM).

When the payload enters the AXI-Stream bus, the hardware no longer asks, "What is the mathematical root of this index?" Instead, it asks the BRAM, "Where does bit 42 go?" The BRAM responds in exactly 1 clock cycle. By unrolling the loop by a factor of 16, we query the BRAM 16 times concurrently, saturating the bus and achieving wire-speed latency.

📊 Trade-offs: Silicon Area vs. Speed

This architecture intentionally trades logic area for pure execution speed. By precomputing the indices, we consume more BRAM tiles on the FPGA die. However, we simultaneously free up Look-Up Tables (LUTs) and DSP slices that would have been wasted on mathematical address generation.

In latency-critical environments like 5G baseband processing, memory space is relatively cheap since the data is predominantly in-flight, but nanoseconds are priceless. Thus, trading BRAM to save nanoseconds is mathematically optimal for the use case.
