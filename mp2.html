Triangular Interleaver (5G-NR)

→ Achieved ~0.41µs latency (~25× faster than baseline)
→ Converted triangular mapping into BRAM-based permutation lookup
→ Parallel processing (16 bits/cycle) with pipelined execution

[problem]

Triangular interleaver requires writing data row-wise into a triangular matrix and reading column-wise.

This leads to: - irregular memory access
- nested loop structure
- high latency in naive implementations

[key insight]

The triangular structure is static and deterministic.

Instead of computing indices at runtime: → precompute the permutation mapping
→ store it in BRAM
→ convert runtime logic into lookup

[design]

- Precomputed permutation table stored in BRAM
- Loop unrolling: 16 elements per cycle
- Fully pipelined architecture
- AXI-based streaming interface

[results]

Latency: ~0.41µs (for 512 bits)
Throughput: 16 elements per cycle
Improvement: ~25× over baseline implementation

🔬 Deep Dive: Why naive approach is slow

Naive implementation directly follows triangular mapping: - nested loops - index computation per element - sequential processing This results in O(N²)-like behavior and high latency.

⚙️ Deep Dive: Lookup-based optimization

By precomputing the mapping: out[i] = in[perm[i]] The problem becomes: - simple memory access - no runtime computation - fully parallelizable This shifts complexity from compute → memory.

📊 Trade-offs

Pros: - drastically reduced latency - simple datapath Cons: - requires BRAM for permutation storage - less flexible for dynamic sizes
